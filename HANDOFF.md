## Що це за проєкт

Веб-застосунок **Balance**, що відкривається як Telegram mini-app.
**Тільки мобільна версія** (ширина ~360–430 px). Мова інтерфейсу — українська.

Трекер: раціон за категоріями обміну продуктів (А–Ж), кроки, тренування,
вага, заміри тіла.

## Стек

React 19 + TS + Vite (збірка в статику) · PHP 8.4 без фреймворку · MySQL 8.0 ·
деплой GitHub Actions → FTP.

Hostia — shared-хостинг: немає Node, SSH і Composer. Laravel свідомо не взято —
~185 МБ вільного місця, vendor/ з'їв би значну частину без користі.

## Головне про модель даних

**Бюджет категорії спільний для всіх її продуктів.** Зберігається щільність
продукту (ккал і білок на 1 г або 1 шт), а не грамівка порції — грамівка
похідна й змінюється протягом дня:

```
ліміт(продукт) = спожито(продукт) + залишок_бюджету_категорії / ккал_на_одиницю
```

Тому з'їдені 52 г овочів знижують ліміт грибів із 600 до 548 г. Підтверджено
скріншотами реального застосунку, закріплено тестом.

**Білок не має цілі** — це лічильник. Натомість діє гарантований нижній поріг:
за будь-якого вибору продуктів користувач набере не менше 50 г. На лімітах
нижче ~1425 ккал бюджет автоматично перерозподіляється в бік білкових категорій.

Математика — `src/lib/ration.ts`, 19 тестів (`npm run test` → `npx vitest run`).

## Робочий процес

Кожен push у `main` автоматично запускає деплой (`.github/workflows/deploy.yml`):
типи → тести раціону → тести підпису Telegram → збірка → перевірка, що секрети
не потрапили у збірку → заливка по FTP.

Перед комітом завжди:
```
npx tsc -b --noEmit && npx vitest run && php api/tests/auth_test.php && npm run build
```

**`api/config.php` НЕ в git і НЕ заливається деплоєм** — лежить тільки на сервері
й локально. Містить пароль БД і токен бота. Перед кожним комітом варто
перевіряти, що він не потрапив у diff.

## Локальна розробка

```
npm --prefix Balance-website run dev     # http://localhost:5173
```

Без бекенду застосунок працює на демоданих із **помітним банером** угорі —
щоб їх не переплутали зі збереженими.

**Щоб перевірити поведінку з реальним асинхронним API** (деякі баги
відтворюються ТІЛЬКИ так — див. нижче), підніміть копію бекенду на SQLite
в тимчасовій теці, **не чіпаючи бойовий `api/config.php`**:

```
cp -r api/. /tmp/apitest/ && rm /tmp/apitest/config.php
# config.php з dsn=sqlite:..., dev_mode=true
API_DIR=/tmp/apitest php -d extension=pdo_sqlite -S 127.0.0.1:8899 router.php
```
Vite проксує `/api` на `127.0.0.1:8899` (див. `vite.config.ts`).

> SQLite не підтримує `ON DUPLICATE KEY UPDATE`, тож збереження раціону,
> кроків і замірів там падатиме. Тренування (звичайний `INSERT`) працюють —
> на них і тестуйте сценарії закриття модалок.

## Ключові рішення по дизайну (не переглядати без запиту клієнта)

- **Шрифт** — Nunito. **Іконки** — Lucide (шляхи скопійовані в `Icon.tsx`,
  без залежності). Додаючи нову — брати саме з lucide.dev, інакше поїде стиль.
- **Без зеленого в базовій палітрі.** Зелений — лише стан «продукт почато».
- **Заголовок екрана стоїть на місці логотипа**, самого логотипа немає.
- **Шапка не закріплена** — їде разом із контентом.
- **8 категорій (А–Ж)**, відтінки взяті з реального застосунку і приглушені.
  Літера на чипі: у світлій темі біла на всіх, у темній темна на всіх.
- **Фон світлої теми** — градієнт `#EBE9F2 → #FAFAFA` на 40%; темної —
  `#343147 → #020202` на 65%.
- Тема перемикається в Налаштуваннях (Авто / Світла / Темна), вибір
  зберігається в `localStorage`, «Авто» стежить за системою.

## Анімації

Спільна тривалість — **1 секунда**, easing `cubic-bezier(0.22, 1, 0.36, 1)`.
Модалки — 500 мс. Колір іконок таб-бару — 200 мс.

Анімується: великі числа, шкали, стовпчики графіків, кільця прогресу.
Графіки й кільця програються заново **на кожен вхід у вкладку і при відкритті
застосунку**.

## Пастки, на які вже наступали (перевірено на практиці)

1. **Контейнер зі скролом мусить мати внутрішні відступи**
   (`padding: 8px 8px 92px; margin: -8px -8px 0`), інакше тінь карток обрізається.

2. **Маска кільця потребує `closest-side`** — без цього товщина обводки
   виходить майже втричі більшою за задану.

3. **З'єднання з MySQL на Hostia стартує в cp1251.** `charset=utf8mb4` у DSN
   недостатньо: серверний `init_connect` виконується після рукостискання
   і перебиває його. У `connectDatabase` є явний `SET NAMES utf8mb4` — не
   прибирати. Симптом був: сервер віддавав `200` з **порожнім тілом**
   (`json_encode` мовчки повертав `false` на битому UTF-8), фронтенд
   отримував `null` і падав із чорним екраном без помилок у консолі.

4. **API віддає `maxUnits: null`, локальний довідник — `undefined`.**
   Перевірка `!== undefined` пропускала `null`, і `null * scale = 0`
   обнуляв ліміт кожного продукту без власної стелі. Тести на `catalog.ts`
   цього не ловили. Є регресійний тест із `null`.

5. **`onClose` як стрілка в пропсах ламає таймери.** Батьки передають
   `onClose={() => setEditing(null)}` — нова функція на кожен рендер. Якщо
   від неї залежить `useEffect`, його cleanup убиватиме таймер закриття
   модалки при кожному ререндері. На бойовому (async API) це давало
   «модалка більше не відкривається», локально з моком — ні. Тому `onClose`
   тримається в `ref`.

6. **`<input type="range">` не досягає `max`, якщо той не кратний `step`.**
   При `max=53, step=5` повзунок упирається в 50. Тому в модалці кількості
   стоїть `step="any"`, а крок застосовується вручну в `onChange`.

7. **Вкладки не розмонтовуються при перемиканні** (`display: none`), інакше
   кільця й числа губили стан анімації. Через це компонентам, яким треба
   програвати анімацію на вхід, потрібен `ScreenActivationContext`.

8. **CSS: стежити за балансом дужок при точкових правках.** Одного разу
   осиротіла `}` зламала розбір усього файлу після неї — `.modal-close`
   мовчки втратив стилі. Швидка перевірка:
   ```
   node -e "const s=require('fs').readFileSync('src/styles/components.css','utf8');let d=0;for(const c of s){if(c==='{')d++;else if(c==='}')d--}console.log(d)"
   ```

9. **Перевірка анімацій із браузера.** Між двома викликами інструмента минає
   більше секунди, тож замір, початий окремим викликом, застане вже
   завершену анімацію. Весь сценарій (дія + вибірка через
   `requestAnimationFrame`) має виконуватись **в одному скрипті**.

## Відомі обмеження

- **Дані продуктів — реконструкція** зі скріншотів старого застосунку;
  щільності пораховані діленням «ккал порції ÷ грами порції». Ліміти
  розходяться з реальними на кілька відсотків. Правити в `api/seed.sql`,
  потім переімпортувати через phpMyAdmin.
- Сума бюджетів категорій ≈1733 проти заявлених 1766 ккал — клієнт лишив як є.
- **Apple Health недоступний**: mini-app працює у WebView без доступу до
  HealthKit. Кроки — ручний ввід.
- **Профілі, створені до 19.07**, можуть мати зламане ім'я (`Р”С–Р°...`) —
  наслідок пастки №3. Лікується зміною імені або видаленням рядка `users`.
- У БД може лишатись тестовий користувач `telegram_id = 999000001`.

## Що зроблено в останній сесії (19.07)

Розгорнуто на бойовий сервер і пройдено цикл правок дизайну:
заголовок замість логотипа, іконки Lucide, градієнт світлої теми,
перемикач теми, анімації графіків/чисел/модалок, прибрано редагування
профілю з Налаштувань.

Виправлено дефекти, знайдені клієнтом: чорний екран у Telegram (пастка №3),
нулі в усіх продуктах (№4), «модалка не відкривається» (№5), повзунок
не доходив до кінця (№6), невидима лінія цілі на графіку кроків.

## Що далі

Клієнт продовжує давати правки по інтерфейсу. Відкритих технічних
завдань немає.

Можливі напрямки, якщо знадобиться:
- повний список продуктів від клієнта замість реконструйованого;
- редагування довідника продуктів без переливки (проста адмінка);
- видалення записів (зараз є лише `DELETE /workouts`, у UI не виведено);
- перенесення дизайн-системи у Figma (клієнт свідомо відклав).
